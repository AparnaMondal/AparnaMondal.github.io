Why Understanding Design Patterns Matters in Software Development

Introduction

Coding agents generate thousands of lines of code every day — syntactically correct, logically structured, and fast.

But these tools have a fundamental limitation: a knowledge cutoff. They do not know your customer's next requirement, your team's constraints, or how your system needs to evolve two years from now. They generate what you ask for — shaped entirely by how well you ask for it.

What AI cannot do:

  • Understand real production constraints
  • Enforce correct architectural boundaries
  • Guarantee the right processing order in safety-critical systems
  • Know which design will survive scaling
This is the quiet truth about AI-generated code:
It reflects the quality of your thinking — not a substitute for it.

In modern software engineering, code generation is no longer the bottleneck — code structure and maintainability are.

This is where design patterns become essential — not just as coding techniques, but as proven architectural solutions to recurring problems. They help build systems that are scalable, maintainable, and aligned with real-world engineering constraints.

In this blog, we explore this using a 5G NR L2 protocol stack — showing how design patterns map to real systems, not just academic theory.

The Reality of AI-Generated Code

AI-generated code is often:

  • Correct in execution
  • Fast to generate
  • But may be inconsistent in structures for complex software
  • Weak in separation of responsibilities
  • Difficult to evolve in large systems if requirement changes frequently

For example, in a telecom or networking system, an AI might generate layered protocol code that works but does not clearly separate concerns or enforce the correct processing order. This becomes a serious issue when scaling to production-grade systems like 5G NR.

In this project, I generated a simplified version of the L2 layer. It took some effort to refine, but the output serves as a useful skeleton to start a 5G NR L2 design. The code can be found below as L2_NR_DesignPattern.cpp.

Having worked across telecommunications technologies such as WLAN, WiMAX, 3G, 4G, and 5G, I have seen how complex production-scale source code becomes when dealing with real customer deployments.

The goal here is not to highlight how quickly or easily the code was generated using an AI coding agent. Instead, it is to emphasize that the same functional outcome can be achieved in multiple ways through prompts and generation approaches. However, as solution owners, it is critical to gather correct requirements from the customer and structure prompts carefully so that the generated code follows proper design principles.

Without architectural discipline, such systems become:

  • Hard to debug
  • Difficult to extend for new features (e.g., new QoS behavior)
  • Prone to tightly coupled logic
  • Expensive to maintain over time
This is exactly where design patterns transform raw code into engineering-grade architecture.

Architecture Example: 5G NR Protocol Stack Through Design Patterns

Let's examine a real-world-style architecture of the 5G NR protocol stack. Each layer demonstrates how design patterns naturally map to system design.

1. SDAP Layer – Simple Mapping and Responsibility Isolation

SDAP sits above PDCP and is introduced in 5G NR to handle QoS flows instead of traditional bearers.

Its responsibility is very focused:

  • On transmit (Tx): It stamps a 1-byte header with QFI and maps flows to a DRB
  • On receive (Rx): It can trigger reflective QoS mapping using the RQI bit

This is a strong example of single responsibility principle (SRP) — a key design pattern concept in object-oriented design.

If AI-generated code tried to combine SDAP with PDCP logic, it would immediately lose clarity and become difficult to maintain.

2. PDCP Layer – Chain of Responsibility Pattern

The PDCP layer handles:

  • Security
  • Header compression
  • Ciphering and integrity protection

This layer directly maps to the Chain of Responsibility pattern, where each step must occur in a strict order:

  • COUNT assignment happens first (required for ciphering and integrity input)
  • ROHC compression follows
  • PDCP header is built (used later for MAC-I computation)
  • MAC-I is computed over header + plaintext
  • Ciphering is applied
  • Finally, the header is prepended

Each step depends on the previous one.

This is critical: If any step is modified or removed, the entire security chain breaks.

Without a design pattern mindset, AI-generated code might mix these steps or reorder them incorrectly, leading to subtle but serious protocol violations.

3. RLC Layer – Strategy + Factory + State Machine Patterns

RLC is responsible for segmentation and reliable delivery (in AM mode). This layer is a perfect example of multiple design patterns working together:

Strategy Pattern

The same RLC entity supports multiple modes:

  • TM (Transparent Mode)
  • UM (Unacknowledged Mode)
  • AM (Acknowledged Mode)

Each mode is a strategy selected at runtime.

Factory Pattern

The correct RLC strategy is created based on RRC configuration parameters, ensuring proper instantiation without hard-coded logic.

State Machine Behavior

In AM mode, the system maintains strict state variables:

  • VT(A)
  • VT(S)
  • VT(MS)

It also processes STATUS PDUs to advance acknowledgment windows.

Additional compliance rules include:

  • SN assigned once per SDU at enqueue time
  • Segments carry SI (2-bit) and SO (16-bit offset)
  • Rel-16 suspend/resume lifecycle enforcement

Without structured patterns, AI-generated implementations often fail to maintain correct state transitions or reuse logic cleanly.

4. MAC Layer – Observer Pattern in Scheduling

The MAC layer handles scheduling and radio resource management.

It strongly reflects the Observer Pattern:

  • The scheduler acts as the subject (publisher)
  • The UE MAC entity acts as the observer (subscriber)

When a grant is issued:

  • The scheduler "notifies" UE entities
  • UE MAC reacts by building a MAC PDU

Inside the MAC PDU:

  • Control Elements (BSR, PHR) are placed first
  • Data subPDUs follow priority rules

Additional constraints:

  • HARQ supports up to 16 processes per direction
  • Redundancy Version (RV) follows gNB-controlled sequence (0 → 2 → 3 → 1) as defined in TS 38.214

This separation ensures scheduling logic remains independent of UE reaction logic — exactly what the observer pattern enforces.

Why This Matters for AI-Generated Code

AI can generate the below stack following the mentioned design patterns, but the result may still:

  • Mix responsibilities across layers
  • Ignore strict processing order
  • Duplicate logic across components
  • Fail under scaling or feature extension

Design patterns enforce:

  • Predictable structure
  • Clear ownership of logic
  • Extensibility without rewriting core systems
  • Compliance with real-world standards (like 3GPP specs)

Conclusion

AI tools are changing how software is written, but they do not automatically guarantee that it has good architecture.

In production telecom systems, architectural flaws can cause dropped connections, security vulnerabilities, failed certification, and entire system redesigns at enormous cost. AI-generated code without pattern awareness will produce exactly these flaws — functional on the surface, broken underneath.

The 5G NR stack example shows a key truth:

Complex systems are not defined by how much code they contain, but by how well their responsibilities are structured.

Design patterns act as the bridge between:

  • Raw AI-generated code
  • And production-grade software architecture

They are not optional knowledge — they are essential engineering discipline.

AI writes code. Design patterns make it correct, scalable, and maintainable.
